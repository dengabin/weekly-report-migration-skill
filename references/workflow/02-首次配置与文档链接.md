# 步骤 2：首次配置与文档链接

**本步目标**：确保 `config.json` 含有效文档链接；成员由 otl 自动填充。

---

## 2.1 何时执行本步

**前置条件**：用户已明确要**执行迁移**（如说「周报迁移」），而非仅「加载 skill」。

| 条件 | 动作 |
|------|------|
| 无 `config.json` | AskQuestion 收集链接 → Agent 写入 config |
| 有 config 但缺 `link_id` | 补问链接 |
| config 完整 | 快速核对链接仍有效，跳过问答 |

## 2.2 AskQuestion 收集项（按序）

1. **组内周报** — 云文档链接（`.otl`）**或**用户直接粘贴的 Markdown（`# 日期` + `## 姓名`）  
   粘贴时 Agent 写入 `.cache/team-report.md`
2. **部门周报链接**（`.ksheet`）  
   例：`https://365.kdocs.cn/l/DEPT_LINK_ID`
3. **四级部门子表名**（仅预检无法解析时）  
   从 Agent 给出的子表列表中选择
4. **小组组名**（仅 `resolve_team_name.py` 退出码 3 时）  
   部门表内多个组、且 otl 成员姓名无法唯一对应到同一组时，**必须**询问；能对应则**不问**（见 [team-name-resolution.md](../team-name-resolution.md)）

## 2.3 不要问用户

- ❌ 成员名单
- ❌ 周次（用户未指定时）
- ❌ WPS_SID（预检通过前不问）
- ❌ 组名（当 otl 姓名已在部门表唯一对应到同一组时）

## 2.4 写入 config 后

Agent 从模板或已有 config 合并用户链接，再执行：

```bash
python scripts/extract/extract_otl_weekly.py --markdown .cache/team-report.md --output .cache/extracted.json
python scripts/plan/build_config_from_extracted.py
python scripts/plan/resolve_team_name.py
```

`resolve_team_name.py` 根据成员姓名在部门子表中反推组名；`status: resolved` 则自动写入 `team_name`，**不问用户**。

## 2.5 本步完成标志

- `config.json` 含 `team_report.link_id`、`dept_report.link_id`（或已有 `.cache/team-report.md`）
- `members[]` 已从 otl 生成
- `team_name` 已解析，或已明确需用户补充组名（暂停在 AskQuestion）
