# 步骤 8：校验与汇报

**本步目标**：确认写回正确，向用户汇报结果。

---

## 8.1 自动校验项

| 检查项 | 标准 |
|--------|------|
| 目标格内容 | 与 `extracted.json` 的 `content` 一致 |
| hypersublink 数量 | 与 patch 前原文一致 |
| 非目标 zip 条目 | 字节级未变（zip 补丁脚本内置） |
| 成员覆盖 | ready 人数与计划一致 |

## 8.2 向用户汇报

应包含：

1. **文档链接**与**云端版本号**（若已上传）
2. **周次**、**子表名**
3. **成功 N 人**、跳过/失败列表及原因
4. 提醒用户 **Ctrl+F5** 强刷浏览器查看
5. 请用户抽查：本周列换行、📄 链接列可点、历史周列未变

## 8.3 失败处理

- **不要**全量覆盖重试
- 仅对失败成员重跑步骤 5–7
- `wps_sid` 过期 → 回步骤 3 §3.3

## 8.4 清理本地缓存（默认）

写回云端**成功**后，除非用户明确要求保留，否则**必须**清理 `.cache/`：

```bash
python scripts/workflow/cleanup_cache.py
```

或在写回时由 `apply_migration.py` **自动**执行（默认行为，无需单独调用）：

```bash
python scripts/workflow/apply_migration.py --upload
# 成功后自动 cleanup
```

| 场景 | 是否清理 |
|------|----------|
| 上传成功，用户未说保留 | ✅ 仅清理 `.cache/` |
| 用户说「清理缓存」 | ✅ 仅清理 `.cache/`（**不删** `config.json`、`auth.yaml`、`vendor/wps365-read`） |
| 用户说「保留缓存」「不要清理 .cache」 | ❌ 加 `--keep-cache` |
| 仅预览、未写回 | ❌ 不清理（后续还要 apply） |
| 写回失败或校验失败 | ❌ 不清理（便于重试/排查） |
| 用户明确要求删除 `config.json` | 单独删除，**不属于**「清理缓存」 |

清理范围：`.cache` 下全部文件与子目录，**保留空 `.cache` 目录**。

向用户汇报时可一句带过：「本地中间文件已清理。」

## 8.5 本步完成标志

- 用户已知迁移结果
- `.cache` 已按上表处理
- 全流程结束
